# debug

- 项目是推荐用户下一个喜欢看的电影，而不是是简单的推荐下一个电影。因为数据集里面由显式的评分数据，而不是只是点击交互。所以输入序列**可以包含所有交互历史**，但监督信号一定**只来自高分**。
- 特征有哪些？user 侧：包含了user_id，gender，age，occupation，zip_code；movie侧：movie_id，genres，isAdult，startYear，popularity，averagerating，多模态embedding。
- 最大序列长度？从原始的数据来看，用户平均历史交互长度为165左右，p75大概是200多，p90大概400，超过150条历史的用户大概占35%。过滤掉非高分（小于等于4）交互movie后，平均长度变成95，p75大概124左右，超过150条正反馈的用户大概占19%。过滤掉低分（小于3）交互movie后，平均长度变为138，p75为177，p90为328。现在召回侧的max_len设置为500，排序侧为300
- 指标不太行 -> 召回策略有问题，不够强，很多物品没被召回。 -> 召回集是所有用户共享的还是**用户有各自的候选集**？或者就是召回质量差
- 数据集不包含天然负样本，因此排序模型也需要hard negative（一开始用in_batch negative太简单了）。召回本来就需要负采样，但召回可以选（in_batch_negative）
- 多路召回，怎么融合多路召回的结果的？按recall率分配各自该召回的数量。多路包括two-tower，GRU4Rec，itemcf，还有基于启发式的popular，genre召回。itemcf做了时间加权，还加入了多模态embedding召回（使用VIT-L14对poster，title，description进行了embedding，user embedding用mean pooling之后的最近序列的movie的embedding来代表，然后内积打分）。最终召回的候选集应该排除掉测试集之前的所有见过的movie，否则指标天然高。当前召回率**recall@200 0.749**，单路最高是GRU4Rec，0.68；然后是itemcf，0.64；然后是双塔，0.57
  - GRU4Rec是序列模型，根据输入序列（每个位置是movie id的embedding），通过GRU（重置门控制上一个阶段的隐状态有多少要保留到这阶段作为信息，更新们控制当前阶段的隐状态用多少当前阶段的信息进行更新）为每个位置生成一个代表当前状态的hidden states，然后可以跟ground truth做监督学习。loss是next item的pairwise的BPR（一对正负样本），可以选listwise的softmax作为辅助，也就是拉开user embedding与正样本的embedding的点积与其他负样本点积的差距。（注，这里还是用prefix吧，因为用户太少，序列不够，绝对样本少了）
  - 双塔模型loss不怎么掉？本来双塔模型的loss就很难降，因为loss主要来自于用户的embedding与item的embedding的内积，而且两个塔之间中间互补交互，梯度很难传递。并且如果负采样很随机，每个item和user的内积都差不多，logits区分度不够，正负样本的logits差不多，无法区分哪些是真正应该召回。debug：设置temperature=0.05，区分开正负样本的logits。（为什么温度设置这么低，因为正则化之后，内积范围为[0,1]，所以logits很小，温度小一点才能起到很大的放缩作用）。并且严格过滤负样本，不允许出现用户交互过的序列，被in_batch设置为负样本。随机负采样是大头，可以加少量相似样本；还可以根据流行度采样，被采样概率Q **正比于** 点击次数的0.75次方。注意，如果非随机采样了，logits要做logQ修正。
  - itemcf是怎么处理的？不是简单做全局共现矩阵，而是结合了用户行为序列的顺序信息。对每个用户的历史交互序列，在局部窗口内统计 item 到 item 的转移关系，比如当前电影前面 10 个历史电影都会和它建立关联，距离越近贡献越大，同时会用评分作为权重，评分越高的交互贡献越大。这样得到的是一个带方向的 item-to-item 转移表。推理时取用户最近 50 个历史 item 作为 anchor，
  - 召回模型不能选择低分movie作为负样本！！！随机的，没见过的负样本才是金负样本，hard negative也只是银。因为这个负采样是为了模拟全库计算，越随机越好。召回的目的是快速筛选出用户可能感兴趣的，用户交互过的movie对召回来说信号已经很强了（即使是低分），这说明这个movie是已经曝光给用户，并且用户点击了，足够作为召回的正样本。只是在排序模型中可以作为负样本。从每个 anchor 查出相似 item，然后按 anchor 的时间位置做衰减加权，最近的历史兴趣权重大一些，衰减函数用的是 1 / log2(rank + 1)，最后把多个 anchor 召回到的 item 分数累加排序。
- 排序模型主要使用DIN这个序列模型，输入所有交互历史的embedding（包括交互的movie的静态特征，用户打的评分以及相对时间gap），用户的静态特征embedding以及候选movie的embedding；loss是sampled softmax而不是BCE，因为最终的目标是优化召回回来的movie的排序，而不是CTR。正样本选择大于等于4分的样本，只拿这个正样本做训练，低分样本可作为负样本。loss按评分加权。负样本可以是热门但未正反馈的物品，随机未被正反馈的物品，但如果不是随机采样，就要进行LogQ修正。
- 早停是需要根据验证集计算出的一个指标来判断的，比如随机采样的ce loss（优点是解耦），或者直接在召回的候选集进行top-k排序，然后看NDCG@20进行早停。

- 序列数据划分时，严格按照时间来划分（滑动窗口），因为用户兴趣与时间强相关，随意划分会导致离线结果很好（因为看到了用户最后时间的兴趣），但在线结果往往不太行。
- 排序模型的训练不依赖于召回模型召回的候选集，但推理过程依赖召回模型的候选集。注意：考虑到排序模型负采样过容易会导致AUC虚高，所以排序模型训练的负采样可以借用召回模型得到的一些item，或者选择其他强的负样本。还有loss可以不用原始的CTR预测（也就是pointwise的BCE，这样的loss直观性将正负样本分开，不关心正样本之间的顺序，因此NDCG不太行），而是listwise/pairwise的loss如BPR还区分了正负样本的先后顺序，因此更能跟最终评测保持一致，因为最终的任务是在召回的候选集中进行top-k预测，指标是NDCG，需要排序能力。
- 判断训练是否收敛？一看训练loss是否稳定了，验证loss是否开始升高（适用于排序，召回并不太使用适用）二是通过梯度的L2范式，只求稳定，不求近似0，考虑到神经网络是非线性非凸的优化问题，且由于SGD，一般梯度很难达到0。三是早停，根据验证时的最终指标如NDCG@20这种，看到基本可以了就可以早停。
- 本项目的离线实验（也就是测试集）是leave-one-out的，排序模型在召回的item库中进行评测，而不是全库评测。全库评测是在验证集上进行的，目的是为了调参，控制DIN早停。指标会比测试集上对召回集评测高很多。（全库AUC，GAUC并无意义。HR@20大概**0.33**左右，NDCG@20大概**0.18**左右；召回集上AUC **0.792**，GAUC **0.80**，HR@20大概**0.280**左右，NDCG@20大概**0.138**左右（HR@10大概**0.199**，NDCG@10大概**0.104**））
- 数据要记得划分训练集，验证集和测试集。召回的验证集用来调多路召回的权重分配，测试集用来做最终的recall@200指标计算（根据训练集加验证集的数据进行推理得到topk个召回的物品）；排序的训练集用来生成正负训练样本，训练DIN、验证集用来调参，利用全库（或近似全库）上计算得到的AUC，GAUC，Recall，NDCG等控制DIN早停，测试集在召回的item库上计算最终的NDCG，Recall，AUC，GAUC等。

- 长序列处理问题：双塔模型输入长序列（500），建模全局兴趣（所有序列embedding的mean pooling，可加近期序列embedding的mean pooling，一起concat）；GRU4Rec输入短序列（150左右），建模短期兴趣变化；itemcf选最近50个交互历史，根据这些交互的movie对其他movie的scores（由历史序列决定，当前的movie会对之后一个window内的movie计分，然后计算score[A][B] = rating_A * rating_B / distance），然后基于距离衰减加权计算一个全局的global score（大小为全局movie数），最后根据这个全局分数取topk个召回的样本。DIN是MoE的思想，有多个不同的兴趣专家，每个专家是一个池化后的embedding（长期兴趣专家是全局mean pooling；短期兴趣专家是最近几个交互序列的attention weighted pooling；高分正反馈兴趣专家是高分样本的rating weighted attention；语义相关兴趣专家是用双塔训练完的item embedding，然后计算候选与序列中movie的内积相似度，取topk个，然后进行attention pooling），然后门控输入candidate item embedding与user profile embedding。测试时分长序列用户和短序列用户。
